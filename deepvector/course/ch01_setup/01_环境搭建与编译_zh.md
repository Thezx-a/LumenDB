## 什么是向量数据库？

在我们写任何代码之前，先来搞清楚一个根本问题：**我们到底在做什么？**

### 传统数据库回顾

你可能用过 MySQL、PostgreSQL 这样的**关系型数据库**（relational database）。它们把数据组织成"行和列"的表格，用 SQL 语言查询。比如你要查"所有年龄大于 25 岁的用户"，数据库会在 `age` 这一列上做精确匹配——25 就是 25，不差一点。

这叫做**精确查询**（exact query），传统数据库非常擅长。

### 但 AI 时代需要不一样的查询

2023 年以来，大语言模型（LLM，比如 GPT、Claude、Llama）彻底改变了软件开发。这些模型有一个核心能力：**把任何东西变成向量**（vector）。

什么是**向量**？简单说，就是一串数字。比如：

- 一张猫的照片 → `[0.23, -0.15, 0.87, 0.04, ...]`（可能有 1024 个数字）
- 一段文字"今天天气真好" → `[0.12, 0.45, -0.33, 0.78, ...]`（可能有 1536 个数字）
- 一段音频 → `[0.56, -0.22, 0.11, 0.93, ...]`（可能有 2048 个数字）

这些数字串叫做**嵌入**（embedding）。它们不是随便生成的——模型经过海量数据训练后，学会了把"语义相似"的东西映射到数字空间中"距离相近"的位置。比如"今天天气真好"和"今天阳光明媚"的向量会很接近，而和"量子力学很复杂"的向量会相距很远。

### 向量数据库要解决的问题

现在问题来了：我有 1 亿张图片的向量，用户上传了一张新图片，我想找"最相似的 10 张"——怎么做？

**暴力搜索**（brute force）：把新向量和 1 亿个向量逐一比较距离。太慢了，可能要几分钟。

**向量数据库**（vector database）就是为了解决这个问题而生的。它会提前把这些向量"整理"成一种特殊的数据结构（叫做**索引**，index），让搜索速度从 O(n) 降到接近 O(log n)。就像图书馆的分类系统——你不用翻遍整个图书馆去找一本关于机器学习的书，直接去"计算机科学→人工智能→机器学习"那个架子就行了。

### 向量数据库 vs 传统数据库

| | 传统数据库（MySQL） | 向量数据库（DeepVector） |
|---|---|---|
| **数据** | 行和列（结构化） | 向量（非结构化：图片、文字、音频的数字表示） |
| **查询** | 精确匹配（WHERE age > 25） | 相似度搜索（找到与查询向量最接近的 K 个向量） |
| **索引** | B+ 树、哈希表 | HNSW、IVF、PQ 等近似最近邻（ANN）算法 |
| **典型应用** | 电商订单、用户信息 | 推荐系统、图像检索、RAG（检索增强生成） |
| **结果** | 精确的，100% 正确 | 近似的（approximate），但速度快几个数量级 |

**RAG**（Retrieval-Augmented Generation，检索增强生成）是目前最火的应用场景之一：用户提问时，先从向量数据库中检索相关文档，然后把文档和问题一起交给 LLM 生成回答。这样 LLM 就不会"胡说八道"了，因为它有了真实的参考资料。

而 **DeepVector** 就是一个用 C++ 实现的、高性能的向量数据库引擎。我们这一整个课程，就是从零开始理解它、构建它、使用它。

## 1.2 克隆仓库与 Git 子模块

```bash
git clone --recurse-submodules https://github.com/Thezx-a/DeepVector.git
cd DeepVector
```

### 什么是 Git 子模块？（Submodule）

**子模块**（submodule）是 Git 的一项功能，允许你在一个 Git 仓库中引用另一个 Git 仓库的**特定提交**。它解决的问题是："我的项目依赖另一个库，我想精确控制使用哪个版本，同时保持两个仓库的独立历史。"

```
DeepVector（主仓库）
├── vendor/MiniKV  → 指向 MiniKV 仓库的 commit abc123
└── vendor/SkyNet  → 指向 SkyNet 仓库的 commit def456
```

### 为什么用 `--recurse-submodules`？

普通的 `git clone` 只会拉取主仓库的代码，子模块目录 `vendor/MiniKV/` 会是空的！`--recurse-submodules` 告诉 Git："克隆主仓库后，递归地初始化并拉取所有子模块。"

如果忘记加这个参数（这是最常见的 Git 新手错误之一），后续用以下命令补救：
```bash
git submodule update --init --recursive
```

### 项目结构概览

```
DeepVector/
├── include/deepvector/     # 公共头文件（C++ 项目中，头文件是"接口说明书"）
│   ├── collection.h     # 核心集合类：负责向量的增删查
│   ├── config.h         # 全局配置：维度、距离度量类型等
│   ├── distance.h       # 距离计算接口（支持 SIMD 加速）
│   └── hnsw.h           # HNSW 索引：图结构的边和层级管理
├── src/                 # 实现源文件（真正的"干活"代码）
│   ├── collection.cpp   # 集合类的实现
│   ├── distance.cpp     # 距离函数的标量/SIMD 实现
│   ├── hnsw.cpp         # HNSW 图构建和搜索的完整逻辑
│   └── storage.cpp      # 基于 mmap 的持久化存储层
├── tests/               # 测试文件（验证代码正确性的"探照灯"）
│   ├── test_basic.cpp
│   ├── test_distance.cpp
│   └── test_hnsw.cpp
├── vendor/              # 第三方依赖（通过子模块管理）
│   ├── MiniKV/          # 嵌入式 KV 存储
│   └── SkyNet/          # 网络/调度库
├── CMakeLists.txt        # 顶层构建描述文件
└── .github/workflows/    # CI/CD 自动化配置
```

## 1.4 编译产物：静态库 vs 动态库

编译完成后在 `build/` 目录下：

```
build/
├── libdeepvector.a           # 静态库（Release 模式）
├── tests/
│   ├── test_basic         # 测试可执行文件
│   ├── test_distance      # 直接运行即可，不需要额外安装
│   └── ...
├── vendor/
│   └── MiniKV/libminikv.a # 依赖的静态库
└── CMakeFiles/
```

### DeepVector 选择静态库的理由

1. **SIMD 分发的需要**：不同 CPU 支持不同的指令集（AVX2、AVX-512 等）。DeepVector 在编译时根据 `-march=native` 生成针对当前 CPU 优化的代码。如果用动态库，要么必须编译到最低通用标准（损失性能），要么需要维护多个 `.so` 变体。

2. **单进程部署**：向量搜索通常是嵌入式场景（嵌入在 RAG 应用中），只有一个进程使用 DeepVector，不存在多进程共享内存的需求。

3. **避免 ABI 兼容性问题**：C++ 不同编译器、不同标准版本之间的 name mangling 差异是真实存在的痛。静态链接把这个问题压缩到编译时解决，而不是拖延到运行时崩溃。

## 1.5 你的第一个 DeepVector 程序

创建 `myapp/main.cpp`:
```cpp
#include <deepvector/collection.h>
#include <iostream>
#include <vector>

using namespace deepvector;

int main() {
    CollectionConfig cfg;
    cfg.dim = 3;                    // 使用 3 维向量（便于可视化理解）
    cfg.metric = DistanceMetric::L2; // 欧几里得距离作为相似度度量

    Collection coll(cfg, "./data");

    std::vector<float> v1 = {1.0f, 0.0f, 0.0f};
    std::vector<float> v2 = {0.0f, 1.0f, 0.0f};
    std::vector<float> q  = {1.0f, 0.1f, 0.0f};

    coll.add(v1.data());
    coll.add(v2.data());

    auto results = coll.search(q.data(), 2);
    for (auto& r : results) {
        std::cout << "id=" << r.id << " dist=" << r.distance << std::endl;
    }
    return 0;
}
```

> **逐行解释**：
> - `#include <deepvector/collection.h>`：引入 DeepVector 的头文件，让编译器知道 `Collection` 类是什么
> - `CollectionConfig cfg`：创建一个配置对象，设置向量维度（dim=3）和距离度量（L2，即欧几里得距离）
> - `Collection coll(cfg, "./data")`：创建一个集合并，数据存放在 `./data` 目录
> - `coll.add(v1.data())`：把向量 v1 插入集合
> - `coll.search(q.data(), 2)`：搜索与查询向量 q 最接近的 2 个向量
> - `DistanceMetric::L2`：L2 就是欧几里得距离，即两点之间的直线距离

创建 `myapp/CMakeLists.txt`:
```cmake
cmake_minimum_required(VERSION 3.16)
project(MyApp LANGUAGES CXX)
set(CMAKE_CXX_STANDARD 17)
set(CMAKE_CXX_STANDARD_REQUIRED ON)

add_subdirectory(../vendor/MiniKV ${CMAKE_BINARY_DIR}/minikv)
add_executable(myapp main.cpp)
target_include_directories(myapp PRIVATE ../include)
target_link_libraries(myapp PRIVATE deepvector minikv)
```

### CMake 依赖引入方式对比

| 方式 | 工作机制 | 适用场景 | 缺点 |
|---|---|---|---|
| `add_subdirectory` | 把子目录的 CMakeLists.txt 当作自己项目的一部分处理 | 紧耦合的项目组（Monorepo） | 必须把源码放在项目内 |
| `FetchContent` | 在配置阶段从 Git URL 下载源码，然后像 `add_subdirectory` 一样处理 | 从 GitHub 拉取第三方库 | 首次构建需下载 |
| `find_package` | 在系统中搜索已安装的库 | 系统预装的大型库 | 需要预先安装 |
| `ExternalProject` | 在构建时下载并编译外部项目 | 巨型独立项目（如 LLVM） | 语法繁琐 |

DeepVector 使用 `add_subdirectory` 是因为 MiniKV 和 SkyNet 与主项目同仓库、共同开发。

## 1.6 CI/CD 与 GitHub Actions

DeepVector 的 `.github/workflows/ci.yml` 定义了自动化测试流程：

```yaml
name: CI
on: [push, pull_request]
jobs:
  build-and-test:
    strategy:
      matrix:
        os: [ubuntu-22.04, ubuntu-24.04]
        build_type: [Release, Debug]
    runs-on: ${{ matrix.os }}
    steps:
      - uses: actions/checkout@v4
        with:
          submodules: recursive
      - name: Configure
        run: cmake -B build -G Ninja -DCMAKE_BUILD_TYPE=${{ matrix.build_type }} -DENABLE_TESTS=ON
      - name: Build
        run: cmake --build build -j$(nproc)
      - name: Test
        run: ctest --test-dir build --output-on-failure
```
